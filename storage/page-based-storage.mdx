title: Page-Based Storage

HozonDB uses a page-based storage system similar to SQLite, where all data is organized into fixed-size pages. This architecture provides efficient I/O operations and enables features like caching and transaction management.

## Page Structure

Every page in HozonDB is exactly **4096 bytes (4KB)** in size, defined by the `PAGE_SIZE` constant.

```rust
pub const PAGE_SIZE: usize = 4096; // 4KB pages
pub type PageId = u32;
```

### Page Types

HozonDB uses different page types for different purposes:

- **Page 0**: Database header (reserved)
- **Page 1**: System catalog storing table metadata
- **Pages 2+**: Table data pages with metadata

## Page Metadata

Each data page (page 2 and above) contains metadata at the beginning that tracks the page's state:

```rust
pub struct PageMetadata {
    pub is_full: bool,        // Whether the page can accept more rows
    pub last_offset: usize,   // Next available byte offset for data
    pub num_rows: usize,      // Number of rows stored in this page
    pub next_page: Option<u32>, // Link to next page (for multi-page tables)
}
```

### Metadata Layout

The first **9 bytes** of each data page are reserved for metadata:

```
Byte 0:       is_full flag (1 byte)
Bytes 1-2:    last_offset (2 bytes, u16 little-endian)
Bytes 3-4:    num_rows (2 bytes, u16 little-endian)
Bytes 5-8:    next_page (4 bytes, u32 little-endian)
```

Defined by these constants:

```rust
pub const PAGE_METADATA_SIZE: usize = 9;
pub const PAGE_DATA_START: usize = 9;  // Data starts after metadata
```

### Usable Data Space

With 9 bytes reserved for metadata, each page has:

- **Total size**: 4096 bytes
- **Metadata**: 9 bytes
- **Available for data**: 4087 bytes

## Page Manager

The `PageManager` struct handles all page operations:

```rust
pub struct PageManager {
    file: Mutex<File>,           // Database file handle
    lock_path: PathBuf,          // Path to lock file
    num_pages: u32,              // Total number of pages
    first_free_page: Option<PageId>, // Head of free page list
}
```

### Core Operations

#### Allocating Pages

```rust
pub fn allocate_page(&mut self) -> io::Result<PageId>
```

When allocating a new page, the PageManager:

1. Checks if there are free pages in the free list (reuses freed pages)
2. If no free pages, extends the database file by one page
3. Initializes metadata for data pages (pages 2+)
4. Returns the new page ID

#### Reading Pages

```rust
pub fn read_page(&self, page_id: PageId) -> io::Result<[u8; PAGE_SIZE]>
```

Reads exactly 4096 bytes from the file at the calculated offset:

```rust
let offset = (page_id as u64) * (PAGE_SIZE as u64);
```

#### Writing Pages

```rust
pub fn write_page(&mut self, page_id: PageId, data: &[u8]) -> io::Result<()>
```

Writes data to a page and syncs to disk. Data shorter than PAGE_SIZE is zero-padded.

#### Freeing Pages

```rust
pub fn free_page(&mut self, page_id: PageId) -> io::Result<()>
```

Adds a page to the free list (LIFO linked list) for reuse.

## Page Initialization

When a new data page is allocated, it's initialized with default metadata:

```rust
fn init_page_metadata_buffer(page_data: &mut [u8; PAGE_SIZE]) {
    page_data[0] = 0;  // is_full = false
    page_data[1..3].copy_from_slice(&9u16.to_le_bytes());  // last_offset = PAGE_DATA_START
    page_data[3..5].copy_from_slice(&0u16.to_le_bytes());  // num_rows = 0
    page_data[5..9].copy_from_slice(&0xFFFFFFFF_u32.to_le_bytes());  // next_page = None
}
```

## Concurrency Control

HozonDB uses file-based locking to prevent concurrent access:

- When opening a database, creates a `.lock` file
- If lock file exists, returns `WouldBlock` error
- Lock is automatically released when `PageManager` is dropped

```rust
fn acquire_lock(lock_path: &Path) -> io::Result<()>
fn release_lock(lock_path: &Path) -> io::Result<()>
```

## Example: Page Lifecycle

```rust
// Create a new database
let mut pm = PageManager::new("mydb.hdb")?;

// Allocate a new page
let page_id = pm.allocate_page()?;  // Returns page 1

// Write data to the page
let data = b"Hello, HozonDB!";
pm.write_page(page_id, data)?;

// Read it back
let page_data = pm.read_page(page_id)?;

// Update metadata
let metadata = PageMetadata {
    is_full: false,
    last_offset: 9 + data.len(),
    num_rows: 1,
    next_page: None,
};
pm.update_page_metadata(page_id, &metadata)?;

// Free the page when no longer needed
pm.free_page(page_id)?;
```

## Performance Characteristics

- **Fixed-size pages** enable predictable I/O operations
- **Page-level caching** can be implemented efficiently
- **Direct offset calculation** for O(1) page access
- **LIFO free list** for fast page reuse
